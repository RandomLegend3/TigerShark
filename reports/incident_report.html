We can make some educated guesses about which stream might be the command channel and which might be the control channel. Let's go over this in detail. Overall, the volume of data and frames is relatively low and consistent across intervals, with most intervals having 20 frames and 1210 bytes of data exchanged. There is a large spike in the interval 8400 <> 8700 seconds, where 3330 frames and 404079 bytes are exchanged.

Typical behavior for a command channel usually involves periodic, low-volume communications. Beaconing is typically associated with the command channel as it involves regular, automated check-ins from system to server. The check-ins are usually small, concise messages sent at periodic intervals to indicate that the compromised system is active. This matches the consistent low-volume data (20 frames, 1210 bytes) intervals observed here, suggesting that the intervals outside the spike likely represent the command channel. The primary purpose of beaconing in the command channel is to maintain an open line of communication, ensuring that the compromised system can receive instructions, or commands from the C2 server.

The control channel, on the other hand, is used for more data-intensive tasks such as exfiltrating large amounts of data or transferring sizeable payloads. The communication over this channel is less frequent but involves larger data packets, differing from the frequent and lightweight nature of beaconing. The significant spike in the 8400 <> 8700-second interval could indicate an interactive session or a more intensive data transfer, such as file uploads/downloads or real-time command execution, indicative of the control channel.

As you can see, visualizing the two TCP streams together in a graph can be highly beneficial. While I could still determine from looking at the two TCP streams separately that it was C2, and which stream was which channel, it wasn't as strikingly obvious as it was when visualizing them together in a graph. NetShareEnumAll via srvsvc lists all shared resources on a server (which could house critical GPO files). This activity could be a precursor to finding and manipulating GPO-related files. Samr functions seen are used to gather information about users, groups, and domain configs. Knowledge gained from samr protocol could guide attackers in targeting specific GPOs. Between srvsvc and samr, the attacker can identify shared resources and gather information on the domain structure, users and groups. Svcctl functions StartServiceW and DeleteService suggest manipulation of services. An attacker could use GPOs to deploy a malicious service across the domain. For example, modifying a GPO to include a startup script that installs or starts a service on all domain controllers . The StartServiceW and DeleteService calls in svcctl could be part of this process. These match the initial Qakbot DLL hash value - efweioirfbtk.dll, umtqqzkklrgp.dll, ltoawuimupfxvg.dll, which just so happen to be listed with the domain controller. Be cautious if you're going to export these files. If you're on Mac or Linux, export them and run a hash, and look the hash up in your favorite malware database. I highly recommend to NOT export those Qakbot DLLs if using Windows OS.
